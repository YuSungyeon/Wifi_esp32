# TX/AP 미구현 상태에서의 동작 가능 범위 정리

## 결론

- **지금도 일부 확인은 가능합니다.**
- 다만, **TX/AP 노드를 ESP32로 직접 운용하는 시나리오 검증은 아직 불완전**합니다.

즉, ESP32 3개 전체 역할(TX/AP + RX1 + RX2)을 100% 재현하려면 TX/AP 펌웨어가 필요합니다.

---

## 왜 일부는 가능한가?

현재 RX 펌웨어와 Mac 수집기는 이미 구현되어 있습니다.

- RX: CSI 수집 + 전처리 + UDP 전송 가능
- Mac: 수신/저장/유실률/장치 상태 점검 가능

RX가 CSI를 받으려면 무선 프레임이 필요하므로, 아래처럼 대체 가능합니다.

1. **기존 공유기(AP)를 TX 역할 대체로 사용**  
2. **노트북/폰 트래픽 발생으로 프레임 공급**

이 경우 RX-Mac 파이프라인(E2E) 검증은 가능합니다.

---

## 지금 당장 가능한 확인 항목

## A. 수집 파이프라인 검증 (가능)

- RX 보드 1~2대에서 CSI 패킷 송신 확인
- Mac에서 `device_id`별 JSONL 저장 확인
- `missing/stale`, `drop_rate` 모니터링 확인
- `session_meta_snapshot.yaml` 저장 확인

## B. 확장성 검증 (가능)

- RX 개수 증가 시 등록표/예상 장치 목록 기반 운영 확인
- 장치별 분리 저장 및 유실 통계 확인

## C. 학습 연결 전 단계 검증 (가능)

- 수집 raw 데이터 품질 확인
- 샘플 수, RSSI, 채널, 시퀀스 누락 확인

---

## 지금 당장 어려운(미완) 확인 항목

## D. TX/AP 포함 3노드 완전 시나리오 (부분 미완)

- ESP32 1대를 TX/AP로 두고 트래픽/채널/세션을 완전히 통제하는 검증
- TX 노드 기준 시간/이벤트 브로드캐스트 연동 검증
- TX-RX 통합 운용 자동화 검증

위 항목은 **TX/AP 전용 펌웨어 구현 후** 완성됩니다.

---

## 현재 권장 동작 흐름 (TX/AP 미구현 상태)

1. **공유기(AP) 준비**  
   - RX와 Mac이 동일 네트워크에 붙도록 설정

2. **Mac 수집기 실행**  
   - `device_registry.csv`, `session_meta.yaml` 기반 실행

3. **RX1/RX2 플래시 및 실행**  
   - 각 보드에 다른 `CSI_DEVICE_ID` 부여(또는 자동 ID)

4. **트래픽 발생**  
   - 노트북/폰 ping, iperf, 스트리밍 등으로 프레임 공급

5. **수집 상태 확인**  
   - `missing/stale/drop_rate` 확인
   - 세션 폴더 내 JSONL/메타 스냅샷 생성 확인

6. **5분 고정 데이터 확보**  
   - 정지 상태 데이터로 품질 baseline 확보

---

## TX/AP 펌웨어 구현 후 최종 동작 흐름

1. TX/AP 노드 부팅(채널/세션 정책 고정)
2. RX 노드 CSI 수집/전처리/송신
3. Mac 수집기 저장/모니터링
4. 세션 단위 데이터 검증
5. 학습용 변환(`raw -> .mat`) 및 모델 학습

---

## 판단 기준 (실행 준비)

- **지금 시작 가능**: RX-Mac 파이프라인 검증, 운영 프로세스 검증
- **추가 필요**: ESP 기반 TX/AP 통제 실험 및 완전 자동화 실험

요약하면, 현재 상태는 “핵심 수집 파이프라인 검증 가능, TX/AP 통합 실험은 후속 구현 필요” 단계입니다.

# WiSLAR CSI 프로젝트 중간개발보고서 (심화)

본 문서는 `examples`를 제외하고, 현재 저장소의 실제 프로젝트 자산(운영 코드 + 학습 코드 + 문서 + `esp32s3_* /build` 산출물)을 포함한 심화 관점 보고서입니다.

## 1. 서론

### 1.1 과제 배경 및 필요성
- 본 프로젝트는 카메라/웨어러블 없이 Wi-Fi CSI(Channel State Information)만으로 실내 동작을 인식하는 시스템을 구축하기 위해 추진되었습니다.
- 기존 기술 대비 차별점:
  - 비가시광/프라이버시 친화형 센싱(영상 미수집)
  - 저비용 ESP32-S3 기반 다중 노드 구성
  - 현장 수집(raw JSONL) -> 학습 입력(MAT)까지 이어지는 데이터 파이프라인 내재화
- 기존 저장소는 오프라인 학습(Temporal U-Net) 중심이었고, 본 과제는 실장비 수집/운영 체계를 결합해 실사용 가능성을 높였습니다.

### 1.2 과제 목표
- 최종 결과물 주요 기능:
  - TX/AP 노드: SoftAP 생성 + UDP broadcast heartbeat 송신
  - RX 노드: CSI 수집 + 경량 전처리 + UDP 송신
  - Mac 수집기: 스키마 검증 + 장치별 JSONL 저장 + 상태지표(missing/stale/drop_rate) 출력
  - 변환기: JSONL -> `train_data.mat`, `test_data.mat` 자동 생성
  - 실시간 MVP: 2클래스(정지/동작) 규칙 기반 콘솔 판정
- 목표 성능 지표(제안):
  - 패킷 유효성: invalid packet 비율 1% 이하
  - 운영 안정성: expected device 기준 missing/stale 0 유지(정상 운영 구간)
  - 수집 품질: 장치별 drop_rate 평균 1% 이하
  - 변환 성공률: 세션별 JSONL -> MAT 변환 성공률 100%

## 2. 설계 및 구현 현황

### 2.1 시스템 아키텍처
- 전체 구성:
  - `esp32s3_tx_ap_node` -> RF 트래픽 공급
  - `esp32s3_csi_sender`(RX N대) -> CSI 전처리/전송
  - `mac_collector` -> 수신/검증/저장/상태 모니터링
  - `data_tools` -> 라벨/변환 자동화
  - `train.py`, `test.py`, `unet/*` -> 학습/평가
  - `realtime` -> 실시간 규칙형 판정
- 데이터 흐름:
  1. TX/AP heartbeat 송신
  2. RX CSI callback 수집 및 전처리
  3. UDP 전송(헤더+payload)
  4. Mac 수집기 JSONL 저장
  5. labels/session-subject 입력 후 MAT 변환
  6. U-Net 학습/평가 또는 실시간 MVP 판정

### 2.2 세부 설계 사항
- 데이터 저장 설계:
  - `mac_collector_output/raw/YYYYMMDD/session_<id>/device_<id>.jsonl`
  - `session_meta_snapshot.yaml` 동시 저장
- 패킷 스키마 설계:
  - `magic/version/header_len/payload_type/session_id/device_id/seq/timestamp/sample_count/csi_amp[]`
- 알고리즘 설계:
  - RX: I/Q -> amplitude -> moving average -> z-score -> clipping
  - Realtime MVP: frame energy 기반 baseline 차이 score -> threshold 이진 판정
- DB 설계:
  - 별도 RDBMS 없음, 파일 기반(raw JSONL + YAML + CSV + MAT) 파이프라인
- UI/UX:
  - 별도 GUI 없음, CLI/로그 중심 운영 인터페이스

### 2.2-1 비전공자 이해를 위한 핵심 개념
- CSI(Channel State Information): Wi-Fi 신호가 공간/사람 움직임에 따라 달라지는 상태값
- TX/AP: 신호를 지속 발생시키는 기준 장치(송신원 + 접속 지점)
- RX: 신호를 관측해 특징을 추출하고 서버로 전송하는 장치
- Collector: RX에서 온 패킷을 검증/저장하는 수집기
- Session: 한 번의 실험 단위(장치 설정/환경/메타데이터를 함께 묶음)

### 2.2-3 하드웨어 배치/기울기 상세

#### 공간 기준
- 방 크기: `3m x 3m`
- 좌표계: `(0,0)=좌하단`, `x`는 오른쪽, `y`는 위쪽
- 설치 높이 기준: `1.2m`

#### 장치 배치도(현재 기준)
- TX/AP(지정): `(1.5, 0.1)`, `orientation_deg=90`
- RX1(101): `(0.2, 0.2)`, `orientation_deg=45`
- RX2(102): `(2.8, 0.2)`, `orientation_deg=135`
- RX3(103): `(1.5, 2.8)`, `orientation_deg=225`

#### 배치 의도
- TX/AP를 하단 중앙에 배치해 상단 RX3와 좌/우 RX로 신호 분산
- RX를 삼각형 형태로 배치해 단일 경로가 아닌 다중 경로 변화를 관측
- 각도값(`orientation_deg`)은 안테나 정면 기준 방향을 수치화해 재현성 확보

#### 운영 시 주의
- 장치 위치/각도 변경 시 같은 `session_id`를 재사용하지 않음
- 변경 내용은 `session_meta.yaml`의 `operator.notes`에 즉시 기록
- 동일 비교 실험은 반드시 동일 좌표/높이/각도로 반복

### 2.2-4 설계 선택의 기술적 근거(Why)

#### 왜 UDP를 기본 전송 프로토콜로 선택했는가?
- 본 시스템의 1차 목표는 "연속 CSI 스트림 확보"이며, 실시간성 저하를 최소화해야 합니다.
- TCP는 재전송/흐름제어 특성상 지연과 버퍼링이 증가할 수 있어, 초기 운영 안정화 단계에서는 불리합니다.
- UDP 선택 후 `seq` 기반 손실 측정(`drop_rate`)을 병행해 품질을 정량 관리하고, 임계 초과 시 TCP/ACK 전환이 가능하도록 설계했습니다.

#### 왜 TX/AP heartbeat(주기 broadcast) 방식을 썼는가?
- CSI를 안정적으로 얻으려면 일정한 무선 트래픽 공급원이 필요합니다.
- 별도 공유기/외부 트래픽 의존을 줄이기 위해 TX/AP가 스스로 트래픽을 생성하도록 했습니다.
- `session_id/seq/timestamp`를 포함해 실험 단위 추적과 운영 디버깅을 쉽게 했습니다.

#### 왜 RX 전처리를 이동평균 + z-score + clipping으로 제한했는가?
- MCU 자원 제약(메모리/연산량) 하에서 안정적으로 동작 가능한 경량 처리만 선택했습니다.
- 이동평균: 순간 잡음 완화
- z-score: 프레임 간 스케일 편차 보정
- clipping: 극단치에 의한 후단 불안정 완화
- 복잡한 특징 추출은 정보 손실/연산비용 증가 리스크가 있어, 현 단계에서는 Mac 후처리로 이관했습니다.

#### 왜 파일 기반 저장 구조를 채택했는가?
- 세션 반복 실험에서는 빠른 저장/복구/복제가 중요하며, JSONL+YAML+CSV는 운영 유연성이 높습니다.
- 장치별 파일 분리(`device_<id>.jsonl`)로 장애 분석과 리플레이가 용이합니다.
- `session_meta_snapshot.yaml`을 동봉해 데이터 단독 이동 시에도 실험 조건을 보존합니다.

#### 왜 현재 배치(삼각형 RX + 하단 중앙 TX/AP)를 기준으로 정했는가?
- 특정 장치 편향을 줄이고 공간 전역 변화를 관측하기 위해 다각도 수신 경로가 필요합니다.
- TX/AP 하단 중앙 배치는 상단/좌우 RX에 대한 경로 균형이 좋아, 단일 경로 의존성을 줄입니다.
- RX 삼각형 배치는 LoS + 반사 경로를 동시에 확보해 움직임 민감도를 높입니다.

#### 왜 운영 문서를 단일 매뉴얼 중심으로 재편했는가?
- 현장 운영자는 "문서 탐색"보다 "실행 순서"가 중요합니다.
- 보고성 문서를 줄이고 실행 문서만 남기면 실수 가능성이 줄고 온보딩 시간이 단축됩니다.

## 4. 문제점 및 해결 방안

### 4.1 기술적 애로사항
- 네트워크 설정 불일치로 인한 무수신
- CSI 옵션/빌드 캐시 불일치로 인한 동작 변동
- 다중 RX 운영 시 장치 식별/등록표 불일치 가능성
- 수집-라벨-변환-학습 간 수동 연결 구간 존재

### 4.2 해결 과정 및 계획
- 현재 적용:
  - 단일 배포 매뉴얼 + 체크리스트 정립
  - device registry / session meta 기반 운영 표준화
  - expected device 모니터링으로 운영 상태 가시화
- 향후 계획:
  - 세션 메타 유효성 자동 검사
  - CRC/무결성 강화, 자동 리포트 추가
  - 학습/평가 자동화 스크립트 보강

### 4.3 설계 변경 사항
- 초기 연구 중심 구조에서 실장비 운영 중심 구조로 확장
- 단일 문서 기반 배포 체계로 문서 구조 단순화
- 운영 문서에서 보고성 문서를 제거하고 실행 문서 중심으로 재편
- 통신 전략을 UDP 우선 운영 + 손실 모니터링 기반 보정으로 구체화
- 하드웨어 배치를 좌표/각도 표준값 기반으로 명시해 재현성 우선으로 변경

## 6. 핵심 의사결정 매트릭스(기술 심화)

### 6.1 전송 계층 의사결정

| 기준 | UDP | TCP | UDP+ACK |
|---|---|---|---|
| 지연 특성 | 낮음 | 중~높음 | 중 |
| 구현 복잡도 | 낮음 | 낮음 | 중~높음 |
| 유실 대응 | 애플리케이션 처리 필요 | 내장(재전송) | 선택적 재전송 |
| 현재 적합성 | 높음 | 중 | 중 |

결정:
- 1단계: UDP 채택(실시간성/구현 단순성 우선)
- 2단계: drop_rate 임계 초과 시 UDP+ACK 또는 TCP 전환

### 6.2 신호처리 파이프라인 의사결정

| 단계 | 후보 | 채택안 | 이유 |
|---|---|---|---|
| 원시 변환 | I/Q 유지, amplitude 변환 | amplitude 변환 | 전송/후처리 단순화 |
| 노이즈 완화 | 없음, FIR, 이동평균 | 3-tap 이동평균 | MCU 부하 대비 효과 |
| 정규화 | 없음, min-max, z-score | z-score | 프레임 간 scale drift 완화 |
| 이상치 처리 | 없음, winsorize, clip | clip(-3~3) | 극단치 억제로 안정성 향상 |

### 6.3 저장 계층 의사결정

| 항목 | 파일 기반(JSONL/YAML/CSV/MAT) | DB 기반 |
|---|---|---|
| 초기 구축 속도 | 빠름 | 느림 |
| 실험 추적성 | 높음(세션 디렉토리) | 설계 의존 |
| 대규모 질의 성능 | 낮음 | 높음 |
| 현재 단계 적합성 | 높음 | 중 |

결정:
- 현재는 파일 기반 채택, 이후 Parquet/DB 확장 계획

## 7. 트레이드오프 및 한계

### 7.1 선택의 대가
- UDP 채택 대가: 손실 가능성 증가 -> `seq/drop_rate` 모니터링으로 보완
- 경량 전처리 채택 대가: 정보 일부 손실 가능 -> 원시 데이터 병행 저장/후처리 계획
- 파일 기반 채택 대가: 대규모 집계 한계 -> 후속 분석 계층 추가 예정

### 7.2 기술적 한계(현 시점)
- 라벨 생성 품질이 사람 입력 품질에 영향을 받음
- 학습 자동화 수준이 완전하지 않아 수동 단계가 일부 존재
- 실시간 판정은 규칙형 MVP로, 모델 추론 성능과 직접 동일시 불가

## 9. 심사 대응 Q&A(기술/비기술)

### Q1. 왜 "완전 무결성"보다 "실시간성"을 우선했나?
- 본 과제의 1차 산출물은 "현장 동작 가능한 수집 파이프라인"입니다. 무결성 강화는 2차 고도화 단계로 분리해 리스크를 단계적으로 관리했습니다.

### Q2. 배치를 수치화해서 관리하는 이유는?
- CSI는 공간 구조에 매우 민감합니다. 위치/높이/각도 수치가 없으면 동일 실험 재현이 거의 불가능합니다.

### Q3. 왜 TX/AP를 별도 노드로 분리했나?
- 트래픽 공급원을 분리하면 RX 수집 안정성이 높고, 외부 AP 환경 변화에 덜 영향을 받습니다.

### Q4. 왜 지금 DB를 도입하지 않았나?
- 현재는 반복 실험과 디버깅 속도가 중요해 파일 기반이 더 효율적입니다. 데이터량 확대 시 DB/Parquet를 도입할 계획입니다.

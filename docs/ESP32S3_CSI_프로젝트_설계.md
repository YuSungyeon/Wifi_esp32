# ESP32-S3 x3 + MacBook CSI 프로젝트 설계안

## 1) 목표와 범위

- **목표**: 3m x 3m 실내 공간에서 ESP32-S3 3대를 이용해 CSI(Channel State Information)를 수집하고, 디바이스에서 1차 전처리 후 MacBook으로 전송하여 저장/학습 가능한 파이프라인 구축
- **범위**:
  - ESP32-S3: CSI 수집 + 경량 전처리 + 패킷 전송
  - MacBook: 수신 서버 + 데이터 저장 + 후처리/시각화/학습

---

## 2) 시스템 전체 구조

### 구성 노드

- **Node A (TX/AP 역할)**: 고정 송신 역할(트래픽 발생 기준점)
- **Node B (RX1)**: CSI 수집 + 전처리 + Mac 전송
- **Node C (RX2)**: CSI 수집 + 전처리 + Mac 전송
- **MacBook (Collector)**: RX 노드 데이터 동기 수집/저장/분석

> 실무적으로는 CSI 안정성을 위해 RX 2대를 주 수집기로 쓰고, TX는 트래픽 발생용으로 분리하는 방식이 운영이 쉽습니다.

### 데이터 흐름

1. TX 노드가 주기적으로 Wi-Fi 프레임 트래픽을 발생
2. RX 노드(B/C)가 CSI 프레임 획득
3. RX 노드에서 1차 전처리(정규화/필터/윈도우 통계) 수행
4. 전처리 결과 + 메타데이터를 MacBook으로 전송
5. MacBook이 수신 데이터 저장(원본/전처리본 분리), 세션 단위 인덱싱

---

## 3) 3m x 3m 방 배치 설계

## 권장 배치

- TX(Node A): 한쪽 벽 코너(높이 1.2m 내외)
- RX1(Node B): 반대편 벽 코너(높이 1.2m)
- RX2(Node C): 측면 벽 중앙(높이 1.2m)
- 측정 대상(사람): 방 중앙 1m x 1m 작업 영역

## 배치 원칙

- **가시선(LoS) + 비가시선(NLoS) 경로**가 동시에 생기도록 배치
- 벽/가구 반사 경로를 일부 활용하되, 매 실험 동일 배치 유지
- 안테나 방향/높이 고정(재현성 확보)
- 각 노드 전원 품질 동일화(노이즈 변동 최소화)
- RX 확장 시 각 노드의 물리 위치/높이/방향을 좌표로 기록(확장 재현성 확보)

---

## 4) ESP32-S3 펌웨어 역할 분리

## Node A (TX/AP)

- SoftAP 또는 고정 송신 노드로 동작
- 일정 주기로 트래픽 발생(ping/UDP beacon/커스텀 frame trigger)
- 실험 세션 ID 브로드캐스트(선택)

## Node B/C (RX)

- CSI 콜백에서 raw CSI 수신
- 프레임 유효성 체크(RSSI, 길이, 서브캐리어 유효 범위)
- 1차 전처리 후 패킷화
- UDP 또는 TCP로 Mac 전송

---

## 5) 1차 전처리(ESP32-S3 내) 정의

MCU에서 가능한 가벼운 처리만 수행하고, 무거운 연산은 Mac으로 넘깁니다.

## 권장 전처리 항목

- **결측/이상치 제거**: 비정상 CSI frame drop
- **진폭 변환**: 복소수 CSI → amplitude
- **간단한 정규화**:
  - 프레임 단위 min-max 또는 z-score(경량 버전)
- **노이즈 완화**:
  - 3~5 tap 이동평균
- **윈도우 특징 추출(선택)**:
  - 평균, 분산, 에너지, 서브캐리어별 표준편차

## ESP에서 보내는 데이터 포맷(권장)

- `session_id`, `device_id`, `seq`, `timestamp_us`
- `rssi`, `channel`, `rate`, `noise_floor(가능 시)`
- `csi_amp[]` 또는 `feature_vec[]`
- `label(optional)` (실험 중 수동 태깅 시)

---

## 6) MacBook 수집 서버 설계

## 수신 계층

- Python 수집기(`asyncio` + UDP/TCP)
- 디바이스별 소켓/버퍼 분리
- sequence 기반 유실률 모니터링

## 저장 계층

- 원본 저장: `raw/{date}/{session}/{device}.jsonl|parquet`
- 전처리 저장: `proc/{date}/{session}/{device}.parquet`
- 세션 메타: `session_meta.yaml` (배치, 사람 수, 동작 프로토콜)

## 동기화

- NTP 수준 동기화 + 수신 시각 보정
- 분석 단계에서 윈도우 정렬(예: 100ms 단위 리샘플)

## N대 확장 설계 원칙

- 수집기는 `device_id`를 key로 완전 동적 처리(장치 수 고정 가정 금지)
- 디렉터리 규칙은 `session_id/device_id` 중심 유지
- 운영 시 기대 RX 목록을 선언하고, 미수신/끊김(stale) 상태를 주기적으로 점검
- 장치 등록표를 별도로 유지:
  - `device_id`
  - 보드 STA MAC
  - 설치 위치(좌표)
  - 펌웨어 버전

---

## 7) 통신 프로토콜 선택 가이드

- **초기 개발 단계**: UDP 권장
  - 장점: 구현 간단, 저지연
  - 단점: 유실 가능
- **데이터 무결성 우선 단계**: TCP 또는 UDP+ACK

권장 순서:
1. UDP로 end-to-end 동작 먼저 완성
2. 유실률 측정
3. 필요 시 재전송/ACK/압축 적용

---

## 8) 실험 프로토콜(어떻게 진행할지)

## 단계 0: 준비

- ESP-IDF 개발환경 구성
- 각 노드 고정 IP 또는 mDNS 이름 부여
- 방 배치 좌표를 문서화(사진 + 치수)

## 단계 1: 단일 링크 검증

- TX 1대 + RX 1대로 CSI 수집 성공 확인
- 5분 고정 상태 데이터 수집(정지/무동작)
- 노이즈 바닥, 유실률, 프레임 속도 측정

## 단계 2: 3노드 통합

- TX + RX2대 동시 수집
- Mac 수집기에서 device별 데이터 분리 저장 확인
- timestamp/seq 기반 정합성 체크
- 기대 RX 목록 대비 missing/stale 점검

## 단계 3: 전처리 검증

- ESP 전처리 on/off 비교 수집
- Mac에서 분포/스펙트럼 비교(정보 손실 확인)
- 파라미터 고정(필터 길이, 정규화 방식)

## 단계 4: 라벨링 실험

- 동작 클래스 정의(예: 배경 + 6동작)
- 동작별 반복 횟수, 사람 수, 위치 조건 표준화
- 세션별 메타데이터 기록(시간, 피험자, 배치)

## 단계 5: 학습 데이터셋 구축

- 윈도우 분할(예: 192 샘플 또는 1~2초)
- train/val/test 분리(사람 기준 분리 권장)
- 클래스 불균형 점검 및 보정

## 단계 6: 모델 학습/평가

- 기존 `train.py`/`test.py` 파이프라인에 맞는 포맷으로 변환
- 베이스라인 모델 학습 후 AP/정확도 비교
- 세션/사람/위치 변화에 대한 일반화 성능 점검
- RX 개수 변화(2대/3대/4대+)에 따른 성능 민감도 비교

---

## 9) 초기 일정(2주 예시)

- **1~2일차**: ESP 펌웨어 기본 송수신 + Mac 수신기 스켈레톤
- **3~4일차**: CSI 파싱/저장, seq/timestamp 검증
- **5~6일차**: ESP 1차 전처리 탑재, on/off 비교
- **7~9일차**: 3m x 3m 표준 시나리오 데이터 수집
- **10~12일차**: 라벨 정제, 데이터셋 포맷 통일
- **13~14일차**: 베이스라인 학습 및 1차 리포트

---

## 10) 리스크와 대응

- **시간 동기화 오차**: sequence + 수신시각 보정, 후처리 정렬
- **무선 간섭(2.4GHz 혼잡)**: 채널 고정, 실험 시간대 고정
- **프레임 유실**: 패킷 크기 축소, 송신 주기 조절, UDP 재전송 정책
- **재현성 저하**: 배치/높이/안테나 방향 문서화 및 고정

---

## 11) 바로 시작할 최소 구현(MVP)

1. RX 노드 1대에서 CSI amplitude만 추출해 UDP 전송  
2. Mac에서 JSONL로 저장 + 실시간 로그 출력  
3. RX 2대로 확장 후 device_id 기반 병합 저장  
4. 이동평균 + z-score만 ESP에 넣고 성능/품질 비교  
5. 동작 2~3개 클래스 소규모 수집 후 학습 파이프라인 연결

이 MVP가 안정화되면 라벨링 프로토콜과 실험 규모를 확장하는 방식이 가장 빠릅니다.

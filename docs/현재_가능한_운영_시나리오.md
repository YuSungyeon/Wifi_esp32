# 현재 가능한 운영 시나리오 (최신 반영)

이 문서는 현재 저장소 기준으로 **실제로 운영 가능한 전체 흐름(수집 -> 변환 -> 학습 입력 생성)**을 정리합니다.

---

## 1) 현재 구현 상태 요약

## 구현 완료

- TX/AP 노드
  - `esp32s3_tx_ap_node` (SoftAP + UDP broadcast heartbeat 송신)
- RX 노드
  - `esp32s3_csi_sender` (CSI 수집 + 1차 전처리 + UDP 전송)
- Mac 수집기
  - `mac_collector/udp_collector_mvp.py`
  - 패킷 검증, JSONL 저장, `missing/stale/drop_rate` 모니터링
  - `device_registry.csv` 자동 로드, `session_meta` 스냅샷 저장
- 데이터 변환기
  - `data_tools/jsonl_to_mat.py`
  - raw JSONL을 `data/train_data.mat`, `data/test_data.mat`로 변환
  - 사람 기준 분할(`session_subject.csv`) 지원
- 운영 문서/템플릿
  - `mac_collector/device_registry.csv`, `mac_collector/session_meta.yaml`
  - `실장비_값_입력_체크리스트.md`, `TX_AP_노드_적용_가이드.md`, `data_tools/README.md`

## 아직 미구현(후속)

- 라벨링 자동 생성/자동 정합성 검사 도구
- 수집->라벨->변환->학습까지 원클릭 자동화 파이프라인
- TCP/ACK 모드, CRC 강제 검증, 자동 리포트 고도화

---

## 2) 현재 운영 가능한 기본 시나리오

## 시나리오 A: ESP32 3대 + MacBook (권장)

- TX/AP 1대: 트래픽 공급 및 채널 고정
- RX 2대: CSI 수집/전송
- Mac: 수신/저장/상태 모니터링
- 후처리: JSONL -> MAT 변환

결과:
- 수집 결과 JSONL 확보
- 학습 입력 `.mat` 생성 가능
- 기존 `train.py`/`test.py`가 읽을 데이터 준비 가능

---

## 3) 운영 절차 (한 세션 기준)

## 단계 0. 준비

1. `mac_collector/device_registry.csv` 실장비 값 입력
2. `mac_collector/session_meta.yaml` 세션 값 입력
3. `실장비_값_입력_체크리스트.md` 점검

## 단계 1. Mac 수집기 실행

```bash
python "mac_collector/udp_collector_mvp.py" \
  --host 0.0.0.0 \
  --port 9999 \
  --output-dir "mac_collector_output" \
  --device-registry-csv "mac_collector/device_registry.csv" \
  --session-meta "mac_collector/session_meta.yaml"
```

기대 로그:
- expected device ID 로드
- 수신 전 `missing/stale` 표시

## 단계 2. TX/AP 노드 실행

`esp32s3_tx_ap_node` 빌드/플래시 후 실행

기대 로그:
- SoftAP started
- heartbeat `seq` 증가

## 단계 3. RX 노드 실행

`esp32s3_csi_sender`를 각 보드에 플래시
- RX1: `CSI_DEVICE_ID=101`
- RX2: `CSI_DEVICE_ID=102`

기대 결과:
- 수집기에서 각 `device_id` packet 증가
- `missing/stale` 해소

## 단계 4. 데이터 수집

- 5분 정지 베이스라인 수집
- 동작 프로토콜 수집

생성 확인:
- `mac_collector_output/raw/YYYYMMDD/session_<id>/device_<id>.jsonl`
- `session_meta_snapshot.yaml`

## 단계 5. JSONL -> MAT 변환

```bash
python "data_tools/jsonl_to_mat.py" \
  --raw-root "mac_collector_output/raw" \
  --output-dir "data" \
  --window-size 192 \
  --stride 96 \
  --labels-csv "data_tools/labels.csv" \
  --session-subject-csv "data_tools/session_subject.csv" \
  --test-subject-ids "S03"
```

생성 확인:
- `data/train_data.mat`
- `data/test_data.mat`

## 단계 6. 실시간 자세(2클래스) 확인

```bash
python "realtime/realtime_pose_mvp.py" \
  --host 0.0.0.0 \
  --port 9999 \
  --baseline-frames 100 \
  --score-window 20 \
  --motion-threshold 0.02 \
  --print-every-sec 1
```

확인 포인트:
- `label=0`(정지), `label=1`(동작) 실시간 출력
- 장치별 `score`, `drop_rate` 동시 확인

## 단계 7. 학습 준비 확인

- `train.py`가 요구하는 키 존재 확인:
  - `train_data_amp`, `train_label_instance`
  - `test_data_amp`, `test_label_instance`(또는 `test_label_mask`)

## 4) 확장 운영 시나리오 (RX N대)

현재 구조는 RX N대를 지원합니다.

운영 규칙:
- 신규 장치는 `device_registry.csv`에 먼저 등록
- `session_meta.yaml`의 `expected_device_ids` 갱신
- 각 RX에 고유 `CSI_DEVICE_ID` 부여(또는 자동 ID)
- 동일 수집기 명령으로 운영 가능

관찰 포인트:
- `missing/stale` 즉시 탐지
- 장치별 `drop_rate` 비교

---

## 5) 현재 가능한 것 / 제한 사항

## 가능한 것

- TX/AP + RX + Mac E2E 수집 운영
- 다중 RX 운영 감시 및 세션 메타 기반 재현성 운영
- 수집 결과를 학습 입력 `.mat`로 변환

## 제한 사항

- 라벨 CSV(`labels.csv`)는 아직 수동 준비가 필요
- 학습 실행 자동화는 아직 미구현(변환 후 `train.py` 수동 실행)
- 실시간 판정은 현재 규칙기반 MVP(학습모델 교체 전 단계)

---

## 6) 권장 운영 단위

1. 연결 검증 세션 (2~5분)
2. 정지 베이스라인 세션 (5분)
3. 동작 수집 세션 (프로토콜 기반)
4. 변환/검증 세션 (JSONL -> MAT)
5. 실시간 판정 세션 (규칙기반 MVP)
6. 학습 세션 (`train.py`/`test.py`)

위 5단계를 반복하면, 확장하면서도 품질을 안정적으로 관리할 수 있습니다.

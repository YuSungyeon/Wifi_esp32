# WiSLAR CSI 프로젝트 중간개발보고서 (운영중심)

본 문서는 실제 운영/배포에 직접 사용되는 코드와 문서를 중심으로 작성합니다.  
(`examples` 및 펌웨어 `build` 산출물 상세는 제외)

## 1. 서론

### 1.1 과제 배경 및 필요성
- 본 과제는 ESP32-S3 기반 다중 노드로 CSI를 수집해 실내 동작 인식을 수행하는 운영형 파이프라인 구축을 목표로 합니다.
- 기존 기술 대비 차별성:
  - 카메라 비의존(프라이버시 이슈 완화)
  - 저비용 하드웨어(ESP32-S3)로 확장 가능한 구조
  - 수집/운영/변환/학습 연결을 한 저장소에서 관리

### 1.2 과제 목표
- 최종 기능:
  - TX/AP 운용
  - RX 다중 수집 및 송신
  - Mac 수집/저장/상태 모니터링
  - JSONL -> MAT 변환 및 학습 입력 생성
  - 실시간 MVP(정지/동작)
- 목표 성능 지표(제안):
  - expected device 기준 `missing/stale` 정상 구간 0 유지
  - 장치별 drop_rate 평균 1% 이하
  - 변환 성공률(세션 기준) 100%
  - 단일 세션 데이터 저장 누락 0건

### 2.2-1 비전공자 이해를 위한 용어
- CSI: Wi-Fi 신호가 공간을 지나며 변형된 특성값(움직임이 있으면 값이 변함)
- TX/AP: 기준 Wi-Fi 신호를 계속 발생시키는 송신/접속 지점 장치
- RX: 신호 변화를 측정해 데이터를 보내는 수신 장치
- drop_rate: 전송되어야 할 순번(seq) 중 빠진 비율
- session: 한 번의 실험 묶음(설정, 장치, 시간 정보가 하나로 관리됨)

### 2.2-3 하드웨어 배치(3m x 3m 기준)

좌표계 기준:
- `(0,0)`: 방의 좌하단
- `x`: 오른쪽 방향 증가
- `y`: 위쪽 방향 증가
- `orientation_deg`: 보드 정면 기준 각도(도)

#### TX/AP 배치(본 보고서 기준 지정값)
- TX/AP: `(1.5, 0.1)`, 높이 `1.2m`, 각도 `90deg`
- 설명: 방 하단 중앙에 두고 위쪽으로 신호를 보내 RX1/RX2/RX3에 고르게 도달하도록 설정

#### RX 배치(등록표 기준)
- RX1(`101`): `(0.2, 0.2)`, 높이 `1.2m`, 각도 `45deg`
- RX2(`102`): `(2.8, 0.2)`, 높이 `1.2m`, 각도 `135deg`
- RX3(`103`): `(1.5, 2.8)`, 높이 `1.2m`, 각도 `225deg`

#### 배치 운영 규칙
- 높이는 모든 장치 `1.2m`로 통일해 실험 간 편차를 줄임
- 각도/위치가 바뀌면 `session_meta.yaml`의 `operator.notes`에 반드시 기록
- 동일 프로토콜 비교 실험에서는 배치를 고정

### 2.2-4 주요 설계 선택의 이유(Why)

#### 왜 TX/AP를 `(1.5, 0.1)`에 두었는가?
- 방 하단 중앙 배치는 좌/우 하단 RX와 상단 RX까지 비교적 균등한 경로를 만들기 쉽습니다.
- 한쪽 벽 구석 배치보다 특정 RX 편향(가까운 장치만 강신호) 가능성이 낮아집니다.
- 실험자가 방 중앙에서 동작할 때, TX->인체->RX로 이어지는 반사/산란 경로 변화가 잘 나타납니다.

#### 왜 RX를 삼각형 형태(좌하/우하/상단)로 배치했는가?
- 동일 움직임이라도 위치/각도에 따라 CSI 변화가 다르므로, 단일 RX보다 다중 RX가 안정적입니다.
- 삼각형 배치는 시야각 중복을 줄이고 공간 커버리지를 높여 결측을 줄이는 데 유리합니다.
- 향후 N대 확장 시 기준점(코너+상단 중앙)으로 재현이 쉬운 배치입니다.

#### 왜 높이를 1.2m로 통일했는가?
- 사람 상체/팔 움직임에 민감한 경로를 얻기 쉽고, 바닥 반사만 과도하게 보지 않게 됩니다.
- 설치 높이가 달라지면 신호 패턴이 크게 바뀌므로, 비교 실험에서는 높이 통일이 필수입니다.

#### 왜 TCP가 아닌 UDP를 사용했는가?
- 목적이 "저지연 연속 샘플 수집"이기 때문에, 재전송/혼잡제어로 지연이 늘어나는 TCP보다 UDP가 유리합니다.
- CSI는 시계열 특성상 일부 패킷 손실보다 지연 급증이 더 치명적일 수 있어, 우선 UDP로 파이프라인을 안정화했습니다.
- 손실은 `seq` 기반 `drop_rate`로 관측하고, 허용 범위를 넘으면 TCP/ACK로 전환하는 단계적 전략을 적용합니다.

#### 왜 RX 전처리를 디바이스에서 수행하는가?
- 원시 I/Q 전체를 보내면 대역폭/CPU 부담이 커지므로, `amplitude + 경량 정규화`로 송신량을 줄입니다.
- 장치단 노이즈 완화를 통해 수집기/후처리의 안정성을 높일 수 있습니다.
- 다만 과도한 전처리는 정보 손실 위험이 있어, 이동평균(3tap) + z-score + clipping만 최소 적용합니다.

#### 왜 DB 대신 파일(JSONL/CSV/YAML/MAT) 기반인가?
- 현재 단계의 핵심은 빠른 실험 반복과 재현성 확보이므로, 배포/백업이 쉬운 파일 기반이 운영 효율이 높습니다.
- 장치별/세션별로 경로가 분리되어 디버깅과 후처리 추적이 쉽습니다.
- 이후 규모가 커지면 Parquet/DB 계층을 추가하는 구조로 확장 가능합니다.

## 6. 의사결정 근거표(심사 대응)

### 6.1 통신 프로토콜 선택

| 선택지 | 장점 | 단점 | 채택 여부 | 채택/제외 이유 |
|---|---|---|---|---|
| UDP | 저지연, 구현 단순, 연속 스트림에 유리 | 패킷 유실 가능 | 채택 | 초기 목표가 "실시간 연속 수집"이므로 지연 최소화가 우선 |
| TCP | 무결성/재전송 보장 | 지연/버퍼링 증가 가능 | 보류 | 실시간성 저하 가능성 때문에 2차 단계에서 조건부 도입 |
| UDP+ACK | 무결성과 지연의 절충 가능 | 구현 복잡도 상승 | 후속 | drop_rate 임계 초과 시 단계적 도입 계획 |

### 6.2 배치 전략 선택

| 항목 | 대안 A | 대안 B | 채택안 | 근거 |
|---|---|---|---|---|
| TX/AP 위치 | 코너 고정 | 하단 중앙 | 하단 중앙 | RX 다수에 대한 경로 균형 확보 |
| RX 배치 | 일렬 배치 | 삼각 배치 | 삼각 배치 | 공간 커버리지/경로 다양성 확보 |
| 높이 | 장치별 자율 | 1.2m 통일 | 1.2m 통일 | 비교 실험 재현성 확보 |
| 각도 | 대략 정렬 | 각도 수치화 기록 | 수치화 기록 | 재설치/재실험 시 동일 조건 재현 가능 |

### 6.3 데이터 저장 구조 선택

| 선택지 | 장점 | 단점 | 채택 여부 | 이유 |
|---|---|---|---|---|
| 파일(JSONL/CSV/YAML/MAT) | 빠른 도입, 이식성, 디버깅 용이 | 대규모 질의/집계 한계 | 채택 | 현 단계는 실험 반복과 추적성이 우선 |
| RDBMS/TSDB | 질의/집계 강점 | 구축/운영 복잡 | 보류 | 데이터량 증가 시점에 도입 예정 |
| Parquet 병행 | 분석 효율 상승 | 파이프라인 추가 필요 | 후속 | 수집 안정화 이후 단계적 반영 |

- [ ] TX/AP heartbeat가 지정 주기로 안정 송신
- [ ] expected RX 전부 수신(`missing_devices=[]`)
- [ ] stale 상태 없음(`stale_devices=[]`)
- [ ] 장치별 drop_rate가 기준치(예: 1%) 이내
- [ ] 세션 종료 후 JSONL + session meta snapshot 생성
- [ ] JSONL -> MAT 변환 성공(`train_data.mat`, `test_data.mat`)

## 9. 예상 질의응답(비전공자 대응)

### Q1. 왜 카메라 대신 Wi-Fi를 쓰나요?
- 영상 없이도 움직임 변화를 감지할 수 있어 프라이버시 부담이 낮고, 저조도 환경에서도 동작합니다.

### Q2. 왜 패킷이 일부 유실돼도 괜찮나요?
- 본 과제의 1차 목표는 "연속 신호 흐름" 확보입니다. 일부 유실은 지표로 관리하고, 임계 초과 시 무결성 강화 모드로 전환합니다.

### Q3. 배치를 고정해야 하는 이유는 무엇인가요?
- 위치/각도/높이는 신호 패턴을 직접 바꾸므로, 비교 실험에서는 조건 고정이 필수입니다.

### Q4. 데이터베이스를 안 쓰면 문제가 없나요?
- 현재 단계(실험 중심)에서는 파일 기반이 더 빠르고 추적이 쉽습니다. 규모가 커지면 DB/Parquet 계층을 추가할 계획입니다.
